# Lineage

Hello fellow PL nerds. What is cool about Prism is not any single ancestor but this particularly fruitful combination of ideas cribbed from other languages 1970s until now. I think this is a nifty point in the PL design space, one whose interesting axes (effects, memory, evaluation order) are orthogonal to those of the lambda cube.

Koka, which is heavy inspiration, provided the evidence-passing compilation of effect handlers and the row-typed surface. The Perceus / FP^2 line contributes garbage-free reference counting, drop-guided in-place reuse, and fully-in-place updates. CBPV contributes the value/computation core that makes both of those analyses clean. Prism wires them together so that the same machinery pays off in several directions at once: evidence passing plus the interprocedural flow analysis makes effect-typed stream pipelines fuse so the evidence and state paths allocate zero per-operation cells, while a genuinely escaping effect falls back to a reified free monad whose cost is constant per live continuation rather than zero (a result Haskell needs rewrite rules and Rust needs monomorphization to approach), the Perceus reuse pass turns the record-update / `deriving (Lens)` sugar into in-place pointer writes (so the "lens library" is a syntax rule plus the memory model, not an optic zoo), and the Verse-inspired failure model is recovered entirely from `final ctl` handlers and effect rows with no new core. Holding all of this honest is the triple-backend parity discipline (interpreter as oracle, LLVM and MLIR byte-identical, leak-clean) backed by a Lean model of the core: an executable abstract machine that mirrors the interpreter, proved correct against a substitution semantics and run as a fourth oracle on the compiler's actual core, with machine-checked determinism, progress, and effect-safety theorems. The synthesis, effect handlers compiled by evidence passing, sitting on Perceus/FBIP memory, with optics-by-pattern and a Verse failure layer, all kept byte-identical across three backends and partially verified in Lean, is the contribution that none of the individual ancestors offers on its own.

## Foundations

Call-by-push-value core IR. After parsing and elaboration, prism lowers the surface language into a CBPV core (`src/core/cbpv.rs`), with the split between values (`Value`) and computations (`Comp`) that CBPV makes primitive: thunks (`Thunk`/`Force`), explicit `Bind` sequencing, `Return`, `Lam`/`App`. This is the calculus of Levy, "Call-by-Push-Value: A Functional/Imperative Synthesis" (his thesis, 2001, and the book of the same title, Springer 2004). CBPV's value/computation discipline is what makes the later effect lowering and the Perceus pass tractable: every effect operation and every allocation lives in a syntactically distinguished position.

Lean model of the core. `models/Prism.lean` mirrors `src/core/cbpv.rs` one variant at a time and gives a substitution-based small-step relation plus a determinism metatheorem (`Step.deterministic`), in the standard operational-semantics tradition (small-step reduction in the style of Plotkin's structural operational semantics, and machine-checked language metatheory as in the POPLmark line of work). On top of this, `models/CEK.lean` models the interpreter itself as an environment machine with an explicit continuation stack, runtime closures, and deep multishot handler capture, the CEK machine of Felleisen and Friedman ("Control Operators, the SECD-machine, and the λ-calculus", 1986). The machine is given as a total executable transition function and proved to implement a big-step natural semantics (`bigstep_runs`, a forward simulation), the classic abstract-machine-correctness result; `models/Meta.lean` adds a progress/safety trichotomy and a unique-normal-form corollary, and `models/Dynamics.lean` proves an effect-safety property (an operation makes progress exactly when a handler is in scope). Because the machine is executable, it is also run as a differential oracle: `prism dump core-json` feeds the compiler's real core to the verified machine, which is checked against the interpreter, closing the gap between the proved model and the running compiler. The proof assistant is Lean 4 (de Moura et al.).

ML-family surface syntax and the functional core. The surface language (let, lambda, ADTs, pattern matching, parametric polymorphism, a prelude of `Option`/`Result`/`List`/`Map` combinators in `lib/prelude.pr`) is squarely in the ML / Haskell tradition: Milner's ML (Milner, "A Theory of Type Polymorphism in Programming", JCSS 1978, in the LCF/Edinburgh ML lineage), with surface ergonomics and naming borrowed from OCaml and Haskell. The Koka-flavored surface (offside layout blocks, `fn f(x) = <block>`, `var x := e`, trailing block arguments, `with` flattening, dot-chaining/UFCS, `mask`, `final ctl`, effect rows) follows Leijen's Koka.

## Type system

Haskell/ML-family inference with bidirectional, higher-rank checking. Inference lives in `src/tc/` (`infer.rs`, `subsume.rs`, `context.rs`, `env.rs`). The polymorphic substrate is Hindley-Milner: Hindley, "The Principal Type-Scheme of an Object in Combinatory Logic" (1969), Milner, "A Theory of Type Polymorphism in Programming" (1978), and the algorithm and principal-types theorem of Damas and Milner, "Principal Type-Schemes for Functional Programs" (POPL 1982). On top of that, prism uses bidirectional typing to handle rank-N polymorphism and subsumption (the `subsume.rs` instantiation of existentials, checking vs. synthesis). The bidirectional discipline traces to Pierce and Turner, "Local Type Inference" (POPL 1998 / TOPLAS 2000), and prism follows specifically the complete and easy higher-rank algorithm of Dunfield and Krishnaswami, "Complete and Easy Bidirectional Typechecking for Higher-Rank Polymorphism" (ICFP 2013). Quantification is predicative, as in that algorithm: a metavariable or type-constructor argument ranges over monotypes, so a `forall` may not instantiate a type parameter (`List(forall a. (a) -> a)` is rejected as impredicative at the annotation), while higher-rank types are free to appear as function arguments, function results, and declared data fields, the positions DK skolemizes and instantiates rather than solves.

Type classes by dictionary passing, with deriving. `class`/`instance` with named instances and `given`/`where` constraints (prelude `Eq`/`Ord`/`Show`, with resolution and dictionary elaboration in `src/tc/classes.rs`, deriving in `src/syntax/desugar/derive.rs`). Constrained functions take hidden dictionary parameters and instances elaborate to ordinary constructor cells. This is the dictionary-passing approach of "How to Make Ad-hoc Polymorphism Less Ad hoc" (POPL 1989), as realized in Haskell. The structural `deriving (Eq, Ord, Show)` mechanism is Haskell's derived instances. Named/explicit instances (`sort_by_ord[ordRev](xs)`) follow the "named instances" / explicit-dictionary line associated with systems like Lean's instance resolution and the "Modular Type Classes" work (Dreyer, Harper, Chakravarty et al., POPL 2007). Classes stay single-parameter but are higher-kinded: a class parameter may range over a type constructor (kind `* -> *`), applied as `f(a)` in method signatures and resolved on the head constructor of an instance head. A class may also declare superclasses (`class Ord(a) given Eq(a)`), whose dictionaries are embedded as the leading fields of the subclass's dictionary cell and projected at a use site, Haskell's superclass-dictionary scheme. On top of this sits the higher-kinded `Functor`/`Applicative`/`Monad`/`Foldable`/`Traversable` tower (Haskell's type-constructor classes, with the applicative structure of McBride and Paterson, "Applicative Programming with Effects", JFP 2008, and the Foldable/Traversable shape of Gibbons and Oliveira, "The Essence of the Iterator Pattern", JFP 2009). The twist is the same "effects instead of monads" substitution the rest of the language makes: `traverse` is effect-polymorphic (`((a) -> b ! {| e}, t(a)) -> t(b) ! {| e}`), so the per-element effect row replaces the `Applicative` parameter and effectful traversal needs no monad and no do-notation.

## Modules and namespaces

Link-time flat namespace with a canonicalizing renamer. A file is a module and a directory is a namespace prefix (`import Data.Map` loads `Data/Map.pr`); a project is a `prism.toml` manifest plus a `src/` tree (`src/project/`), resolved from the project root rather than the entry file's directory. Name resolution (`src/resolve/`) loads the transitively imported modules and rewrites every top-level definition to a canonical module-qualified symbol, exported names as `Data.Map.insert` and private ones as the unforgeable `Data.Map@helper`, then merges all modules into one flat program keyed by those globally unique symbols. This whole-program renamer-then-link model is the Rust module/path system (modules as namespaces, `use`/qualified paths including the full dotted path, `pub` visibility, and `pub use` re-export spelled `pub import`) rather than the ML module calculus of structures, signatures, and functors (Harper and Lillibridge, "A Type-Theoretic Approach to Higher-Order Modules with Sharing", POPL 1994; Leroy, "Manifest Types, Modules, and Separate Compilation", POPL 1994). Because identity is the canonical symbol and not the short name, qualified references reach disjoint namespaces and two modules may export the same name and coexist; the long-term aim is to grow this toward the modules-as-implicits unification of Dreyer, Harper, Chakravarty et al., "Modular Type Classes" (POPL 2007) and White, Bour, and Yallop, "Modular Implicits" (ML Workshop 2014).

Instance coherence by provenance. Instances stay global (every loaded module's instances populate one pool, as in Haskell) but each now records its defining module, so the checker can flag an orphan instance, one defined apart from both its class and its head type, and overlapping instances coming from different modules, while keeping the existing named-instance, ambiguity-at-use model (`f[name]` selection) rather than imposing Haskell's global single-instance coherence. The orphan condition is GHC's orphan-instance rule (an instance must live with its class or its type); the choice to warn rather than forbid overlap, and to resolve ambiguity at the use site, is the modular-implicits reading of coherence as a use-site property. Superclass declarations (`class Ord(a) given Eq(a)`) form a directed graph that must be acyclic: the checker rejects a cycle with a named path at class-construction time, so the dictionary-projection walk that discharges a superclass obligation always terminates.

## Effects

Algebraic effects and handlers. Effect declarations (`effect E { ctl op(T) : R }`), `handle e with` arms, multishot `resume`, and `final ctl` non-resumable clauses are the language's central control abstraction (interpreter `perform` in `src/eval/`, lowering in `src/core/effect_lower/`). The semantic foundation is the algebraic-effects program of Plotkin and Power, "Adequacy for Algebraic Effects" (FoSSaCS 2001) and "Algebraic Operations and Generic Effects" (2003), made programmable by handlers in Plotkin and Pretnar, "Handlers of Algebraic Effects" (ESOP 2009). The pioneering language is Eff (Bauer and Pretnar, "Programming with Algebraic Effects and Handlers", JLAMP 2015), with the row-typed surface form following Koka (Leijen).

Extensible effect rows and row polymorphism. Effect signatures are open rows (`! {E | r}`), inferred and unioned without an umbrella enum, including the structural error rows and the row machinery in `src/types/effects.rs` / `src/tc/`. Row polymorphism descends from Wand, "Complete Type Inference for Simple Objects" (LICS 1987) and Remy's work on records and variants. The specific scoped-labels presentation prism's rows resemble is Leijen, "Extensible Records with Scoped Labels" (TFP 2005), applied to effects as in Koka's effect types (Leijen, "Type Directed Compilation of Row-Typed Algebraic Effects", POPL 2017). Effect-row aliases (`alias Name = {A, B}`) are a purely syntactic convenience over this.

Evidence passing for handler compilation. The fast path that replaces the free-monad reification with direct calls, carrying each active clause as evidence to its perform site, is in `src/core/effect_lower/evidence.rs` (with `thread_ev`/`strip_resume`). This is Xie and Leijen, "Generalized Evidence Passing for Effect Handlers" (ICFP 2021) and the related "Effect Handlers, Evidently" (Xie, Brachthauser, Hillerstrom, Schuster, Leijen, ICFP 2020), as deployed in Koka. The structural detection of tail-resumptive clauses (resume called once, in tail position) and their rewriting to plain returns is the optimization that makes evidence passing pay off. The broader semantics-of-handlers context is Bauer and Pretnar's handler calculus and the work of Biernacki, Pirog, Polesiuk, and Sieczkowski on the semantics and abstraction of handlers (POPL 2018, 2019).

Free-monad fallback for effects. When an effect escapes tracking (buried in data extracted by an unfollowed `case`, dynamically applied, masked, genuinely multishot, or self-referential), lowering falls back to reifying computations as `EPure`/`EOp(id, skip, arg, k)` cells threaded by `ebind` (the `skip` field carries the mask depth, the number of matching handlers the op must still bypass) (`src/core/effect_lower/mod.rs`). This free/freer-monad encoding of effects is Swierstra, "Data Types a la Carte" (JFP 2008), and the extensible-effects line of Kiselyov, Sabry, and Swords, "Extensible Effects: An Alternative to Monad Transformers" (Haskell Symposium 2013), with "Freer Monads, More Extensible Effects" (Kiselyov and Ishii, Haskell 2015).

Constant-stack driving of the reified continuation. A closed handler on the free-monad path is driven, by default, by a single self-tail-recursive loop (`{n}@region`) rather than mutually recursive driver functions: it re-enters itself by a tail call on the resumed continuation, which the backend lowers to `musttail`, so a deep tail-resumptive recursion or a parameter-passing State loop runs in constant native stack instead of growing it per resumption. This is the abstract-machine reading of handlers, trampolining the reified continuation through a tail loop the way the CEK interpreter (`models/CEK.lean`) keeps its continuation on an explicit stack; the bounded-iteration constant-stack property is the one Koka and the evidence-passing line obtain for tail-resumptive handlers (Xie and Leijen), extended here to the parameter-passing State handler by threading its function-typed answer `S -> A` as an accumulator (the defunctionalized-continuation move of Reynolds, "Definitional Interpreters for Higher-Order Programming Languages", 1972, specialized to a single state thread). The reification itself is unchanged, so the per-operation cell cost stays and only the stack discipline improves; the loop reuses the `qApply` continuation queue of the free-monad path, so the freer-monad machinery is the substrate it drives rather than a thing it replaces. Opting out (`PRISM_NATIVE_EFFECTS=0`) restores the mutually recursive driver, and the interpreter oracle's whole-corpus parity holds byte-for-byte either way.

Scoped / masked handlers and forwarding. `mask<Eff>(e)` lets an operation bypass the innermost matching handler, masks nest, and an "open" handler re-emits unhandled ops outward to an enclosing handler. The scoping discipline follows Wu, Schrijvers, and Hinze, "Effect Handlers in Scope" (Haskell Symposium 2014) and Koka's `mask` and handler-stratification scheme (Leijen). The cross-effect forwarding in the native lowering (a driver that returns `Eff` and re-emits the unhandled-op arm with a re-entry continuation) is the handler-forwarding semantics from the same scoped-handlers tradition.

## Memory and evaluation

Perceus reference counting with reuse. Precise reference counting (`Dup`/`Drop` inserted in `src/core/fbip.rs`, runtime `prism_rc_inc`/`prism_rc_dec`, env-gated leak oracle) is Reinking, Xie, de Moura, and Leijen, "Perceus: Garbage Free Reference Counting with Reuse" (PLDI 2021). The drop-guided in-place reuse pass (`reuse` after `insert_rc`: a uniquely owned, dropped scrutinee whose tail rebuilds a same-or-smaller constructor turns its `Drop` into a `ReuseToken` and the rebuild into a `Reuse`, so `map`/tree-map mutate the spine in place) is the reuse half of Perceus, refined in Lorenzen and Leijen, "Reference Counting with Frame-Limited Reuse" (ICFP 2021).

FBIP, fully-in-place functional programming. The discipline that lets functionally-pure code run as in-place mutation when ownership is unique (the pitch behind the record-update sugar `{ r | hit.pos.x = v }` compiling to pointer writes, and `with_<f>` setters being reused) is Lorenzen, Leijen, and Swierstra, "FP^2: Fully in-Place Functional Programming" (ICFP 2023). The same uniqueness-gated mutation extends past inductive spines to a flat heap cell: the growable `Array` is an ordinary reference-counted cell, so `array_set`/`array_push` write in place when the array is uniquely owned (rc 1) and copy otherwise, the functional-array-as-mutation idea checked dynamically by the reference count rather than statically by a uniqueness type.

Beyond the runtime reuse, prism exposes the paper's `fip`/`fbip` as a static discipline checked at compile time (`src/core/fbip.rs`, `src/core/tailrec.rs`). `fbip` proves zero fresh allocation plus a call-graph closure (an annotated body may only call annotated, allocation-free callees); `fip` adds two more of the paper's properties: linearity, checked on the source term so each owned, non-immediate binding is consumed at most once (scalars exempt, since a tagged inc/dec is a no-op, the heap-versus-machine-word distinction the paper draws), and bounded stack, requiring every recursive call in the call-graph SCC to be a tail call or a single tail-modulo-cons/-add, classified by the same `tailrec` module codegen reads to emit the loop, so acceptance never outruns what the backend lowers. This is honestly an analysis-and-reject checker layered on the CBPV core, not FP^2's typed-by-construction calculus with its soundness theorem, and it implements only the zero-allocation `fip(0)` slice without the paper's `fip(n)` allocation budget; the bounded-stack clause and the shared tail/TRMC classifier are prism's own framing of "fully in place" as constant heap and stack both, rather than a transcription of the paper's rules. The honesty net is the independent token-balance verifier (`balanced`) and the `PRISM_CHECK_LEAKS` oracle, plus IR assertions that every accepted `fip` function lowers to a loop. The SCC is built over an over-approximated call graph that unions direct calls with first-class references (a function flowing as a bare value is a potential indirect recursive cycle a call-only graph would miss), which is sound but can pull a function into the tail-recursion group merely for capturing another; a bounded-stack rejection therefore distinguishes a genuine direct-call cycle from a capture-only one in its message, so the user sees why the discipline applies.

Tail-call optimization, and tail recursion modulo cons / destination passing. Self tail calls of equal arity emit `musttail` for constant-stack native loops. A recursive call feeding one constructor field in tail position (`Cons(y, map(f, rest))`) is split into a destination-passing `musttail` loop, with an integer-accumulator variant for `1 + f(x)` (`src/codegen/emit.rs`, with the tail/TRMC classification factored into `src/core/tailrec.rs` so the `fip` bounded-stack check and codegen share one decision). TRMC and destination passing trace to Friedman and Wise, "Unwinding Stylized Recursions into Iterations" (1975) and the constructor-in-tail-position idea (Risch, and the "TRMC" tradition), and most directly to the modern OCaml realization: Bour, Clement, and Scherer, "Tail Modulo Cons" (JFLA 2021 / OCaml 5). The stack-safe interpreter is a flat CEK machine (Felleisen and Friedman's CEK, 1986).

Uniqueness and borrowing. Prism does not expose a uniqueness-type surface, but the ownership model underneath Perceus (function parameters and let-bindings owned and consumed once, closure captures borrowed and dup'd before a consuming use) is the operational realization of the same ideas: the uniqueness types of the Clean language (Barendsen and Smetsers, "Conventional and Uniqueness Typing in Graph Rewrite Systems", FSTTCS 1993), linear logic in programming via "Linear Types Can Change the World!" (1990), and the borrow discipline popularized by Rust. The `var` escape analysis (a closure carrying a block-local var's ops out of scope is rejected) is in the same spirit as borrow-scope checking.

## Optics and pattern matching

Pattern-match compilation. Match arms compile to decision-tree-style dispatch with specialization and default matrices, and a Maranget usefulness matrix drives exhaustiveness and redundancy checking (`src/tc/coverage.rs`, compilation in `src/core/elaborate/`). The decision-tree compilation is Maranget, "Compiling Pattern Matching to Good Decision Trees" (ML Workshop 2008), and the usefulness/exhaustiveness algorithm is Maranget, "Warnings for Pattern Matching" (JFP 2007). The earlier compilation tradition is Augustsson, "Compiling Pattern Matching" (FPCA 1985). Guards are handled the GHC way: a fallible guard leaves its arm out of the usefulness matrix (the guard may fail at runtime, so it neither completes coverage nor shadows a later arm), while an irrefutable `if true` guard counts as a plain wildcard.

View patterns and pattern synonyms (smart deconstructors). `pattern Name(ps) for T = view ... make ...` desugars a match-position use to calling `view` and matching `Some`, and an expression-position use to calling `make`. A class-dispatched form (`pattern First(n) for Peek = view peek`) resolves the view through a class method (`examples/optics.pr`, `examples/class_pattern.pr`, desugar in `src/syntax/desugar/synonyms.rs`). View patterns descend from "Views: A Way for Pattern Matching to Cohabit with Data Abstraction" (POPL 1987). The bidirectional view/make pair (a deconstructor that is also a builder) is GHC's pattern synonyms (Pickering, Erdi, Peyton Jones, Eisenberg, "Pattern Synonyms", Haskell Symposium 2016) and the Scala extractor shape (`unapply`/`apply`).

Lenses and optics. The optic story is deliberately not a profunctor library: it is record-update paths (`{ r | a.b.c = v }`, nested single-constructor rebuilds, FBIP-reused) plus `deriving (Lens)` synthesizing a getter `<f>_of` and setter `with_<f>` per field (`examples/lens_derive.pr`, `examples/optics.pr`). A `pattern` with both `view` and `make` is literally a prism. The optics tradition is Foster, Greenwald, Moore, Pierce, and Schmitt, "Combinators for Bidirectional Tree Transformations: A Linguistic Approach to the View-Update Problem" (POPL 2005 / TOPLAS 2007). The functional-getter/setter encoding is van Laarhoven's representation (the "van Laarhoven lens"), popularized by Kmett's `lens` library, and the prism-as-bidirectional-pattern correspondence is the conceptual through-line.

## First class structures

Stream fusion / deforestation. Streams are effectful producers (`emit`), transformers are re-emitting handlers, consumers are folding handlers (prelude `srange`/`smap`/`skeep`/`stake`/`sfold`/`ssum`/`scollect`), and a fused pipeline allocates no intermediate collection and, on the evidence/state paths, zero `EOp` cells (`src/core/effect_lower/flow.rs` for the interprocedural escape/signature analysis, `src/core/effect_lower/state/` for fold-consumer state threading, gated by allocation-counter stanzas). The fusion goal descends from "Deforestation: Transforming Programs to Eliminate Trees" (TCS 1990). The shortcut/foldr-build form is Gill, Launchbury, and Peyton Jones, "A Short Cut to Deforestation" (FPCA 1993), and the consumer/producer stepper form is Coutts, Leshchinskiy, and Stewart, "Stream Fusion: From Lists to Streams to Nothing at All" (ICFP 2007). Prism's twist is that the fusion is realized by handler inlining and evidence threading rather than by GHC-style rewrite rules.

Verse-inspired failure model. The `Fail` axis (anonymous recoverable `fail()`, `guard`, `optional`/`succeeds`/`default`, `??` fallback, `?.` optional chaining, failable comprehension guards, and `transact body else fallback` for atomic rollback of live `var`s) is built entirely on `final ctl` handlers and the var-as-state machinery. The "failable only in a failure context" guarantee falls out of the effect row for free (an unhandled `Fail` is the ordinary unhandled-effect error). The direct source is the Verse calculus: Augustsson, Breitner, Claessen, Jhala, Peyton Jones, Shivers, Steele, and Sweeney, "The Verse Calculus: A Core Calculus for Deterministic Functional Logic Programming" (ICFP 2023). The reify-to-`Option`/`Bool` consumers are the `Maybe`/`Alternative` idiom (Haskell), and the short-circuiting comprehension-guard behavior echoes goal-directed evaluation as in Griswold and Griswold's Icon.

Structured error handling as effects. `error N(T)` declares a one-op never-resuming effect, `throw`/`try`/`catch` are handler sugar (one nested `final ctl` per arm), and uncaught errors are unhandled-effect type errors naming the open row (desugar in `src/syntax/desugar/effects/`). This is the "exceptions as algebraic effects" view (Plotkin-Pretnar handlers specialized to non-resumptive operations) rather than a built-in exception mechanism. The open, union-without-conversion error rows are the row-polymorphism payoff (Wand / Leijen, as above).

## Backends and verification

Triple-backend parity. The tree-walking interpreter is the oracle, and the LLVM backend (via inkwell) and a text-MLIR backend are byte-identical to it across the whole corpus, both sharing one generic emitter (`src/codegen/emit.rs`, the `Isa` trait, with `llvm.rs`/`mlir.rs` as thin syntax tables). LLVM is Lattner and Adve, "LLVM: A Compilation Framework for Lifelong Program Analysis and Transformation" (CGO 2004). MLIR is Lattner, Amini, Bondhugula, Cohen, et al., "MLIR: Scaling Compiler Infrastructure for the End of Moore's Law" (CGO 2021). Using an independent reference interpreter as a differential oracle for the optimizing backends is a translation-validation / differential-testing practice (Pnueli, Siegel, Singerman, "Translation Validation", TACAS 1998) rather than a feature of any one language.
