# Lineage

Hello fellow PL nerds. What is cool about Prism is not any single ancestor but this particularly fruitful combination of ideas drawn from languages from the 1970s to the present. I think this is a nifty point in the PL design space, one whose interesting axes (effects, memory, evaluation order) are orthogonal to those of the lambda cube.

The organizing idea, though, is a single invariant rather than the pile of borrowings. Every observable behavior of a program is a pure function of its source and its pinned input trace, and nothing below the source, not the effect-lowering tier, the backend, the optimization level, or the thread schedule, may leak into it. Most of what reads like a feature list is a corollary of that one theorem: byte-for-byte parity across three backends, content-addressed definition identity, incremental compilation that recompiles only a change's Merkle closure, deterministic replay, and a bootstrap checkable by diverse double-compilation are five faces of the same guarantee, not five separate mechanisms. The pieces cribbed below are the material; determinism is the frame.

Koka, a major inspiration, provided the evidence-passing compilation of effect handlers and the row-typed surface. The Perceus / FP^2 line contributes garbage-free reference counting, drop-guided in-place reuse, and fully-in-place updates. CBPV contributes the value/computation core that makes both of those analyses clean. Prism wires them together so that the same machinery pays off in several directions at once: evidence passing plus the interprocedural flow analysis makes effect-typed stream pipelines fuse so the evidence and state paths allocate zero per-operation cells, while a genuinely escaping effect falls back to a reified free monad whose cost is constant per live continuation rather than zero (a result Haskell needs rewrite rules and Rust needs monomorphization to approach), the Perceus reuse pass turns the record-update / `deriving (Lens)` sugar into in-place pointer writes (so the "lens library" is a syntax rule plus the memory model, not an optic zoo), and the Verse-inspired failure model is recovered entirely from `never` handlers and effect rows with no new core. Holding all of this honest is the triple-backend parity discipline (interpreter as oracle, LLVM and MLIR byte-identical, leak-clean) backed by a Lean model of the core: an executable abstract machine that mirrors the interpreter, proved to soundly realize a big-step natural semantics (it computes exactly the specified value, and by determinism no other, so it is licensed as the oracle; its agreement with a substitution small-step semantics is checked on worked reductions, not by a general simulation theorem) and run as a fourth oracle on the compiler's actual core, with machine-checked determinism, progress, and effect-safety theorems. The synthesis, effect handlers compiled by evidence passing, sitting on Perceus/FBIP memory, with optics-by-pattern and a Verse failure layer, all kept byte-identical across three backends and partially verified in Lean, is the contribution that none of the individual ancestors offers on its own.

## Foundations

Call-by-push-value core IR. After parsing and elaboration, prism lowers the surface language into a CBPV core, with the split between values and computations that CBPV makes primitive: thunks (`Thunk`/`Force`), explicit `Bind` sequencing, `Return`, `Lam`/`App`. This is the calculus of Levy, "Call-by-Push-Value: A Functional/Imperative Synthesis" (his thesis, 2001, and the book of the same title, Springer 2004). CBPV's value/computation discipline is what makes the later effect lowering and the Perceus pass tractable: every effect operation and every allocation lives in a syntactically distinguished position.

Lean model of the core. The Lean model mirrors the CBPV core one variant at a time and gives a substitution-based small-step relation plus a determinism metatheorem, in the standard operational-semantics tradition (small-step reduction in the style of Plotkin's structural operational semantics, and machine-checked language metatheory as in the POPLmark line of work). On top of this, a CEK model captures the interpreter itself as an environment machine with an explicit continuation stack, runtime closures, and deep multishot handler capture, the CEK machine of Felleisen and Friedman ("Control Operators, the SECD-machine, and the λ-calculus", 1986). The machine is given as a total executable transition function and proved to implement a big-step natural semantics (a forward simulation), the classic abstract-machine-correctness result; a further metatheory adds a progress/safety trichotomy and a unique-normal-form corollary, and an effect-safety property (an operation makes progress exactly when a handler is in scope). Because the machine is executable, it is also run as a differential oracle: the compiler's real core is fed to the verified machine, which is checked against the interpreter, closing the gap between the proved model and the running compiler. The proof assistant is Lean 4 (de Moura et al.).

Independent linting of the compiler's own IR. This is GHC's discipline of a typed intermediate language re-checked by its own linter: System FC, the typed Core of Sulzmann, Chakravarty, Peyton Jones, and Donnelly ("System F with Type Equality Coercions", TLDI 2007), is re-typechecked by Core Lint after each optimization pass, so a miscompiling pass is caught structurally rather than by its output drifting. Prism applies the same idea at two boundaries. Between checking and elaboration the checker's per-node decisions, which constructor a field access resolves to, which dictionary discharges a constraint, are frozen into one dense artifact, the checked HIR, and an independent linting pass re-verifies each stored fact against the live constructor, instance, and class environment before elaboration may consume it. Elaboration then builds a witness-carrying typed Core, and every optimizer, effect-lowering, and reference-counting pass rewrites that Core in typed form while an independent verifier re-checks its scopes, types, instantiations, effects, handlers, and reuse invariants after each pass, before the types are erased for code generation. It is the compile-time twin of the differential oracle above: one holds the compiler honest against its own types, the other against the language's semantics.

ML-family surface syntax and the functional core. The surface language (let, lambda, ADTs, pattern matching, parametric polymorphism, a prelude of `Option`/`Result`/`List`/`Map` combinators) is squarely in the ML / Haskell tradition: Milner's ML (Milner, "A Theory of Type Polymorphism in Programming", JCSS 1978, in the LCF/Edinburgh ML lineage), with surface ergonomics and naming borrowed from OCaml and Haskell. The Koka-flavored surface (offside layout blocks, `fn f(x) = <block>`, `var x := e`, trailing block arguments, `with` flattening, dot-chaining/UFCS, `mask`, `never`, effect rows) follows Leijen's Koka.

## Type system

Haskell/ML-family inference with bidirectional, higher-rank checking. The polymorphic substrate is Hindley-Milner: Hindley, "The Principal Type-Scheme of an Object in Combinatory Logic" (1969), Milner, "A Theory of Type Polymorphism in Programming" (1978), and the algorithm and principal-types theorem of Damas and Milner, "Principal Type-Schemes for Functional Programs" (POPL 1982). On top of that, prism uses bidirectional typing to handle rank-N polymorphism and subsumption (the instantiation of existentials, checking vs. synthesis). The bidirectional discipline traces to Pierce and Turner, "Local Type Inference" (POPL 1998 / TOPLAS 2000), and prism follows specifically the complete and easy higher-rank algorithm of Dunfield and Krishnaswami, "Complete and Easy Bidirectional Typechecking for Higher-Rank Polymorphism" (ICFP 2013). Quantification is predicative, as in that algorithm: a metavariable or type-constructor argument ranges over monotypes, so a `forall` may not instantiate a type parameter (`List(forall a. (a) -> a)` is rejected as impredicative at the annotation), while higher-rank types are free to appear as function arguments, function results, and declared data fields, the positions DK skolemizes and instantiates rather than solves.

Type classes by dictionary passing, with deriving. `class`/`instance` with named instances and `given`/`where` constraints (prelude `Eq`/`Ord`/`Show`, with resolution, dictionary elaboration, and deriving). Constrained functions take hidden dictionary parameters and instances elaborate to ordinary constructor cells. This is the classic dictionary-passing approach to ad-hoc polymorphism, as realized in Haskell. The structural `deriving (Eq, Ord, Show)` mechanism is Haskell's derived instances. Named/explicit instances (`sort_by_ord(xs, using ordRev)`) follow the "named instances" / explicit-dictionary line associated with systems like Lean's instance resolution and the "Modular Type Classes" work (Dreyer, Harper, Chakravarty et al., POPL 2007). Classes stay single-parameter but are higher-kinded: a class parameter may range over a type constructor (kind `* -> *`), applied as `f(a)` in method signatures and resolved on the head constructor of an instance head. A class may also declare superclasses (`class Ord(a) given Eq(a)`), whose dictionaries are embedded as the leading fields of the subclass's dictionary cell and projected at a use site, Haskell's superclass-dictionary scheme. On top of this sits the higher-kinded `Functor`/`Applicative`/`Monad`/`Foldable`/`Traversable` tower (Haskell's type-constructor classes, with the applicative structure of McBride and Paterson, "Applicative Programming with Effects", JFP 2008, and the Foldable/Traversable shape of Gibbons and Oliveira, "The Essence of the Iterator Pattern", JFP 2009). The twist is the same "effects instead of monads" substitution the rest of the language makes: `traverse` is effect-polymorphic (`((a) -> b ! {| e}, t(a)) -> t(b) ! {| e}`), so the per-element effect row replaces the `Applicative` parameter and effectful traversal needs no monad and no do-notation.

## Modules and namespaces

Link-time flat namespace with a canonicalizing renamer. A file is a module and a directory is a namespace prefix (`import Data.Map` loads `Data.Map` from the `Data` directory); a project is a `prism.toml` manifest plus a source tree, resolved from the project root rather than the entry file's directory. Name resolution loads the transitively imported modules and rewrites every top-level definition to a canonical module-qualified symbol, exported names as `Data.Map.insert` and private ones as the unforgeable `Data.Map@helper`, then merges all modules into one flat program keyed by those globally unique symbols. This whole-program renamer-then-link model is the Rust module/path system (modules as namespaces, `use`/qualified paths including the full dotted path, `pub` visibility, and `pub use` re-export spelled `pub import`) rather than the ML module calculus of structures, signatures, and functors (Harper and Lillibridge, "A Type-Theoretic Approach to Higher-Order Modules with Sharing", POPL 1994; Leroy, "Manifest Types, Modules, and Separate Compilation", POPL 1994). Because identity is the canonical symbol and not the short name, qualified references reach disjoint namespaces and two modules may export the same name and coexist.

Instance coherence by canonical designation. Instances stay global (every loaded module's instances populate one pool, as in Haskell) and each records its defining module. Resolution is coherent without Haskell's structural single-instance rule: for each (class, type-head) there is exactly one canonical instance and implicit resolution always selects it. A lone instance is canonical automatically; when two or more share a head, one must be named in a top-level `canonical Class(Head) = name` declaration, and an undesignated overlap is a hard error reported at definition. A different instance is reached only by an explicit use-site override, the trailing `using` argument (`sort_by_ord(xs, using ordRev)`), which changes no other resolution, so meaning never silently depends on which instance the resolver picked. The provenance record also lets the checker warn on an orphan instance, one defined apart from both its class and its head type, which is GHC's orphan-instance rule (an instance must live with its class or its type). The named-instance plus canonical-designation scheme is closer to Lean's instance resolution with default instances, and to the modular-implicits reading of coherence (Dreyer, Harper, Chakravarty et al., POPL 2007; White, Bour, Yallop, ML Workshop 2014), than to Haskell's global coherence. Superclass declarations (`class Ord(a) given Eq(a)`) form a directed graph that must be acyclic: the checker rejects a cycle with a named path at class-construction time, so the dictionary-projection walk that discharges a superclass obligation always terminates.

Content-addressed core. A top-level definition is identified by the hash of its elaborated core rather than by its name. Each definition is hashed over its alpha-normalized core with every free reference to another top-level symbol replaced by that symbol's own hash, so the program becomes a Merkle DAG and a definition's hash transitively commits to everything it calls. This is the model of Unison (Chiusano, Bjarnason, et al.): the codebase as a content-addressed database in which names are a mutable index over immutable `hash -> definition` entries, so a rename is an O(1) metadata edit, two versions of a dependency coexist as two hashes, and an unchanged hash is already compiled. What began as a way to name definitions is now the compiler's spine: the same hash is the cutoff key for incremental checking, the cache key for every durable artifact from checked bodies to native objects, the subject of backend parity, the identity a bootstrap is diverse-double-compiled against, and the record a build reads back to explain why it recompiled anything; content-addressed instance coherence, once an open question, is enforced across programs at commit time. What Prism deliberately does not take is Unison's managed-codebase ambition, a structured store as the source of truth, a codebase editor, and a runtime that ships code by hash: it keeps ordinary source text as the source of truth, so the hash is an identity-and-incrementality device layered over text rather than a replacement for it.

Whole-program synthesis without the whole-program recompile. The link model above merges every module into one flat program, which is whole-program compilation in the sense of MLton: defunctionalize and monomorphize across the entire program, with no separate-compilation boundary left to block a cross-module optimization (Weeks, "Whole-Program Compilation in MLton", ML Workshop 2006, over the flow-directed closure conversion of Cejtin, Jagannathan, and Weeks, "Flow-Directed Closure Conversion for Typed Languages", ESOP 2000). The standing objection to that design is compile time: with no module firewall, a one-line edit can in principle re-analyze the whole program, which is why MLton is an ahead-of-time release compiler rather than an edit-compile-loop one. Prism's answer is to make the merged whole content-addressed rather than monolithic. Because a definition's identity is the hash of its alpha-normalized core with every free reference replaced by the callee's own hash (the Merkle DAG above), an edit moves only the hashes on the path from the changed definition up through its dependents; every hash it leaves fixed names an artifact, a checked interface, an optimized core fragment, a native object, that is by construction already built, so the program is folded together out of pieces the compiler reuses rather than recomputes (the content-addressed store). Whole-program synthesis and incremental rebuild stop being opposites: the program is still compiled as one merged whole, but the merge is a fold over content-addressed pieces, and an unchanged piece carries the proof, its unchanged hash, that its prior output is still valid. The sound importer cutoff is the interface hash specifically: an edit that leaves a definition's externally visible type-and-effect signature fixed keeps its interface hash fixed, so importers are not re-checked even though the body was rebuilt, MLton's discarded separate-compilation firewall recovered as a derived property of content addressing rather than as a module-language construct. This turns the same Unison hash-as-identity device toward the compile-time economics MLton pays for whole-program optimization rather than toward Unison's managed codebase.

Cast as an incremental system, that compiler is a synthesis of three more. The demand-driven, early-cutoff dependency discipline, re-run a query only when a dependency's result actually moved rather than merely its revision, is the red-green algorithm of rustc and the Salsa framework (rust-analyzer). The durable, content-addressed store of build artifacts is Nix's, and Nix's hermeticity, that a hash match is a validity proof, is the determinism contract restated for builds. Identifying a unit by the hash of its normalized code is Unison's, above. What each of those keys "unchanged" on differs, a revision counter in Salsa, a per-session fingerprint in rustc, a derivation hash at whole-package granularity in Nix, and Prism collapses all of it to one content hash at definition and strongly-connected-component granularity over ordinary text, so a single identity serves the incremental cutoff, the artifact cache, the parity subject, and the bootstrap check instead of four keys for four jobs. The conditions under which such a memoized, dependency-tracked build is order-independent, and therefore safe to cache and to parallelize, are Mokhov, Mitchell, and Peyton Jones, "Build Systems à la Carte" (ICFP 2018).

## Effects

Algebraic effects and handlers. Effect declarations (`effect E { op(T) : R }`), `handle e with` arms, multishot `resume`, and `never` non-resumable clauses are the language's central control abstraction. The semantic foundation is the algebraic-effects program of Plotkin and Power, "Adequacy for Algebraic Effects" (FoSSaCS 2001) and "Algebraic Operations and Generic Effects" (2003), made programmable by handlers in Plotkin and Pretnar, "Handlers of Algebraic Effects" (ESOP 2009). The pioneering language is Eff (Bauer and Pretnar, "Programming with Algebraic Effects and Handlers", JLAMP 2015), with the row-typed surface form following Koka (Leijen).

Extensible effect rows and row polymorphism. Effect signatures are open rows (`! {E | r}`), inferred and unioned without an umbrella enum, including the structural error rows and the row machinery. Row polymorphism descends from Wand, "Complete Type Inference for Simple Objects" (LICS 1987) and Remy's work on records and variants. The specific scoped-labels presentation prism's rows resemble is Leijen, "Extensible Records with Scoped Labels" (TFP 2005), applied to effects as in Koka's effect types (Leijen, "Type Directed Compilation of Row-Typed Algebraic Effects", POPL 2017). Checking follows that discipline literally: a function's effect row is made equal to its expected row by pure unification over scoped labels rather than by covariant subsumption, so a narrower row fits a wider context only by solving a row variable and never by silent widening, and a function's own latent row is a quantified variable solved afresh at each use, exactly the scoped-labels unification of Leijen (TFP 2005) as carried into effect rows by Koka (POPL 2017). Effect-row aliases (`alias Name = {A, B}`) are a purely syntactic convenience over this.

Evidence passing for handler compilation. The fast path that replaces the free-monad reification with direct calls carries each active clause as evidence to its perform site. This is Xie and Leijen, "Generalized Evidence Passing for Effect Handlers" (ICFP 2021) and the related "Effect Handlers, Evidently" (Xie, Brachthauser, Hillerstrom, Schuster, Leijen, ICFP 2020), as deployed in Koka. The structural detection of tail-resumptive clauses (resume called once, in tail position) and their rewriting to plain returns is the optimization that makes evidence passing pay off. The broader semantics-of-handlers context is Bauer and Pretnar's handler calculus and the work of Biernacki, Pirog, Polesiuk, and Sieczkowski on the semantics and abstraction of handlers (POPL 2018, 2019).

Free-monad fallback for effects. When an effect escapes tracking (buried in data extracted by an unfollowed `case`, dynamically applied, masked, genuinely multishot, or self-referential), lowering falls back to reifying computations as `EPure`/`EOp(id, skip, arg, k)` cells threaded by `ebind` (the `skip` field carries the mask depth, the number of matching handlers the op must still bypass). This free/freer-monad encoding of effects is Swierstra, "Data Types a la Carte" (JFP 2008), and the extensible-effects line of Kiselyov, Sabry, and Swords, "Extensible Effects: An Alternative to Monad Transformers" (Haskell Symposium 2013), with "Freer Monads, More Extensible Effects" (Kiselyov and Ishii, Haskell 2015).

Constant-stack driving of the reified continuation. A closed handler on the free-monad path is driven, by default, by a single self-tail-recursive loop rather than mutually recursive driver functions: it re-enters itself by a tail call on the resumed continuation, which the backend lowers to `musttail`, so a deep tail-resumptive recursion or a parameter-passing State loop runs in constant native stack instead of growing it per resumption. This is the abstract-machine reading of handlers, trampolining the reified continuation through a tail loop the way the CEK interpreter keeps its continuation on an explicit stack; the bounded-iteration constant-stack property is the one Koka and the evidence-passing line obtain for tail-resumptive handlers (Xie and Leijen), extended here to the parameter-passing State handler by threading its function-typed answer `S -> A` as an accumulator (the defunctionalized-continuation move of Reynolds, "Definitional Interpreters for Higher-Order Programming Languages", 1972, specialized to a single state thread). The reification itself is unchanged, so the per-operation cell cost stays and only the stack discipline improves; the loop reuses the continuation queue of the free-monad path, so the freer-monad machinery is the substrate it drives rather than a thing it replaces. Opting out (`PRISM_NATIVE_EFFECTS=0`) restores the mutually recursive driver, and the interpreter oracle's whole-corpus parity holds byte-for-byte either way.

Scoped / masked handlers and forwarding. `mask<Eff>(e)` lets an operation bypass the innermost matching handler, masks nest, and an "open" handler re-emits unhandled ops outward to an enclosing handler. The scoping discipline follows Wu, Schrijvers, and Hinze, "Effect Handlers in Scope" (Haskell Symposium 2014) and Koka's `mask` and handler-stratification scheme (Leijen). The cross-effect forwarding in the native lowering (a driver that returns `Eff` and re-emits the unhandled-op arm with a re-entry continuation) is the handler-forwarding semantics from the same scoped-handlers tradition.

## Memory and evaluation

Perceus reference counting with reuse. Precise reference counting (`Dup`/`Drop` insertion, runtime increment/decrement, an env-gated leak oracle) is Reinking, Xie, de Moura, and Leijen, "Perceus: Garbage Free Reference Counting with Reuse" (PLDI 2021). The drop-guided in-place reuse pass (a uniquely owned, dropped scrutinee whose tail rebuilds a same-or-smaller constructor turns its `Drop` into a `ReuseToken` and the rebuild into a `Reuse`, so `map`/tree-map mutate the spine in place) is the reuse half of Perceus, refined in Lorenzen and Leijen, "Reference Counting with Frame-Limited Reuse" (ICFP 2021).

FBIP, fully-in-place functional programming. The discipline that lets functionally-pure code run as in-place mutation when ownership is unique (the pitch behind the record-update sugar `{ r | hit.pos.x = v }` compiling to pointer writes, and `with_<f>` setters being reused) is Lorenzen, Leijen, and Swierstra, "FP^2: Fully in-Place Functional Programming" (ICFP 2023). The same uniqueness-gated mutation extends past inductive spines to a flat heap cell: the growable `Array` is an ordinary reference-counted cell, so `array_set`/`array_push` write in place when the array is uniquely owned (rc 1) and copy otherwise, the functional-array-as-mutation idea checked dynamically by the reference count rather than statically by a uniqueness type.

Beyond the runtime reuse, prism exposes the paper's `fip`/`fbip` as a static discipline checked at compile time. `fbip` proves zero fresh allocation plus a call-graph closure (an annotated body may only call annotated, allocation-free callees); `fip` adds two more of the paper's properties: linearity, checked on the source term so each owned, non-immediate binding is consumed at most once (scalars exempt, since a tagged inc/dec is a no-op, the heap-versus-machine-word distinction the paper draws), and bounded stack, requiring every recursive call in the call-graph SCC to be a tail call or a single tail-modulo-cons/-add, classified by the same tail-recursion classifier codegen reads to emit the loop, so acceptance never outruns what the backend lowers. This is honestly an analysis-and-reject checker layered on the CBPV core, not FP^2's typed-by-construction calculus with its soundness theorem, and it implements only the zero-allocation `fip(0)` slice without the paper's `fip(n)` allocation budget; the bounded-stack clause and the shared tail/TRMC classifier are prism's own framing of "fully in place" as constant heap and stack both, rather than a transcription of the paper's rules. The honesty net is the independent token-balance verifier and the `PRISM_CHECK_LEAKS` oracle, plus IR assertions that every accepted `fip` function lowers to a loop. The SCC is built over an over-approximated call graph that unions direct calls with first-class references (a function flowing as a bare value is a potential indirect recursive cycle a call-only graph would miss), which is sound but can pull a function into the tail-recursion group merely for capturing another; a bounded-stack rejection therefore distinguishes a genuine direct-call cycle from a capture-only one in its message, so the user sees why the discipline applies.

Tail-call optimization, and tail recursion modulo cons / destination passing. Self tail calls of equal arity emit `musttail` for constant-stack native loops. A recursive call feeding one constructor field in tail position (`Cons(y, map(f, rest))`) is split into a destination-passing `musttail` loop, with an integer-accumulator variant for `1 + f(x)` (the tail/TRMC classification is factored out so the `fip` bounded-stack check and codegen share one decision). TRMC and destination passing trace to Friedman and Wise, "Unwinding Stylized Recursions into Iterations" (1975) and the constructor-in-tail-position idea (Risch, and the "TRMC" tradition), and most directly to the modern OCaml realization: Bour, Clement, and Scherer, "Tail Modulo Cons" (JFLA 2021 / OCaml 5). The stack-safe interpreter is a flat CEK machine (Felleisen and Friedman's CEK, 1986).

Uniqueness and borrowing. Prism does not expose a uniqueness-type surface, but the ownership model underneath Perceus (function parameters and let-bindings owned and consumed once, closure captures borrowed and dup'd before a consuming use) is the operational realization of the same ideas: the uniqueness types of the Clean language (Barendsen and Smetsers, "Conventional and Uniqueness Typing in Graph Rewrite Systems", FSTTCS 1993), the linear-logic-in-programming tradition, and the borrow discipline popularized by Rust. The `var` escape analysis (a closure carrying a block-local var's ops out of scope is rejected) is in the same spirit as borrow-scope checking.

## Optics and pattern matching

Pattern-match compilation. Match arms compile to decision-tree-style dispatch with specialization and default matrices, and a Maranget usefulness matrix drives exhaustiveness and redundancy checking. The decision-tree compilation is Maranget, "Compiling Pattern Matching to Good Decision Trees" (ML Workshop 2008), and the usefulness/exhaustiveness algorithm is Maranget, "Warnings for Pattern Matching" (JFP 2007). The earlier compilation tradition is Augustsson, "Compiling Pattern Matching" (FPCA 1985). Guards are handled the GHC way: a fallible guard leaves its arm out of the usefulness matrix (the guard may fail at runtime, so it neither completes coverage nor shadows a later arm), while an irrefutable `if true` guard counts as a plain wildcard.

View patterns and pattern synonyms (smart deconstructors). `pattern Name(ps) for T = view ... make ...` desugars a match-position use to calling `view` and matching `Some`, and an expression-position use to calling `make`. A class-dispatched form (`pattern First(n) for Peek = view peek`) resolves the view through a class method. View patterns descend from the classic idea of views, which let pattern matching operate over an abstract data type without exposing its representation, as later taken up in the Haskell/GHC line. The bidirectional view/make pair (a deconstructor that is also a builder) is GHC's pattern synonyms (Pickering, Erdi, Peyton Jones, Eisenberg, "Pattern Synonyms", Haskell Symposium 2016) and the Scala extractor shape (`unapply`/`apply`).

Lenses and optics. The optic story is deliberately not a profunctor library: it is record-update paths (`{ r | a.b.c = v }`, nested single-constructor rebuilds, FBIP-reused) plus `deriving (Lens)` synthesizing a getter `<f>_of` and setter `with_<f>` per field. A `pattern` with both `view` and `make` is literally a prism. The optics tradition is Foster, Greenwald, Moore, Pierce, and Schmitt, "Combinators for Bidirectional Tree Transformations: A Linguistic Approach to the View-Update Problem" (POPL 2005 / TOPLAS 2007). The functional-getter/setter encoding is van Laarhoven's representation (the "van Laarhoven lens"), popularized by Kmett's `lens` library, and the prism-as-bidirectional-pattern correspondence is the conceptual through-line.

## First class structures

Stream fusion / deforestation. Streams are effectful producers (`emit`), transformers are re-emitting handlers, consumers are folding handlers (prelude `srange`/`smap`/`skeep`/`stake`/`sfold`/`ssum`/`scollect`), and a fused pipeline allocates no intermediate collection and, on the evidence/state paths, zero `EOp` cells (the interprocedural escape/signature analysis and fold-consumer state threading, gated by allocation-counter stanzas). The fusion goal descends from deforestation, the classic problem of removing the intermediate structures a pipeline would otherwise allocate. The shortcut/foldr-build form is Gill, Launchbury, and Peyton Jones, "A Short Cut to Deforestation" (FPCA 1993), and the consumer/producer stepper form is Coutts, Leshchinskiy, and Stewart, "Stream Fusion: From Lists to Streams to Nothing at All" (ICFP 2007). Prism's twist is that the fusion is realized by handler inlining and evidence threading rather than by GHC-style rewrite rules.

Verse-inspired failure model. The `Fail` axis (anonymous recoverable `fail()`, `guard`, `optional`/`succeeds`/`default`, `??` fallback, `?.` optional chaining, failable comprehension guards, and `transact body else fallback` for atomic rollback of live `var`s) is built entirely on `never` handlers and the var-as-state machinery. The "failable only in a failure context" guarantee falls out of the effect row for free (an unhandled `Fail` is the ordinary unhandled-effect error). The direct source is the Verse calculus: Augustsson, Breitner, Claessen, Jhala, Peyton Jones, Shivers, Steele, and Sweeney, "The Verse Calculus: A Core Calculus for Deterministic Functional Logic Programming" (ICFP 2023). The reify-to-`Option`/`Bool` consumers are the `Maybe`/`Alternative` idiom (Haskell), and the short-circuiting comprehension-guard behavior echoes goal-directed evaluation as in Griswold and Griswold's Icon.

Structured error handling as effects. `error N(T)` declares a one-op never-resuming effect, `throw`/`try`/`catch` are handler sugar (one nested `never` per arm), and uncaught errors are unhandled-effect type errors naming the open row. This is the "exceptions as algebraic effects" view (Plotkin-Pretnar handlers specialized to non-resumptive operations) rather than a built-in exception mechanism. The open, union-without-conversion error rows are the row-polymorphism payoff (Wand / Leijen, as above).

## Backends and verification

Triple-backend parity. The tree-walking interpreter is the oracle, and the LLVM backend (via inkwell) and a text-MLIR backend are byte-identical to it across the whole corpus, both sharing one generic emitter, with the LLVM and MLIR backends as thin syntax tables over it. LLVM is Lattner and Adve, "LLVM: A Compilation Framework for Lifelong Program Analysis and Transformation" (CGO 2004). MLIR is Lattner, Amini, Bondhugula, Cohen, et al., "MLIR: Scaling Compiler Infrastructure for the End of Moore's Law" (CGO 2021). Using an independent reference interpreter as a differential oracle for the optimizing backends is a translation-validation / differential-testing practice (Pnueli, Siegel, Singerman, "Translation Validation", TACAS 1998) rather than a feature of any one language.
