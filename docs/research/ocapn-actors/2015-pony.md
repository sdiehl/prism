Pony is an actor-based, object-oriented, concurrent language at the intersection of three concerns this survey studies: type-level capability safety (its _deny capabilities_), message-passing concurrency (its _actor model_), and a novel approach to memory management across actor boundaries (its _distributed GC protocol_, ORCA). The ORCA protocol is the most relevant to OCapN, because CapTP's distributed GC is deliberately acyclic, and Pony demonstrates what a cycle-capable alternative looks like and why it is harder.

## Deny Capabilities (Reference Capabilities)

Pony's type system assigns every reference a _capability_ that controls what the holder may do:

| Capability | Aliasable? | Mutable? | Data-race-safe? |
|---|---|---|---|
| `iso` (isolated) | No | Yes | Yes |
| `trn` (transition) | Yes (read-only) | Yes | Yes |
| `ref` (reference) | Yes | Yes | No (must be local) |
| `val` (value) | Yes | No | Yes |
| `box` (boxed) | Yes | Maybe | Yes |
| `tag` (tag) | Yes | No | Yes |

The key insight: a `val` reference (immutable, shared) is safe to copy across actors because no one can mutate it, and an `iso` reference (unique, mutable) is safe to send because the sender loses access on transfer. This is exactly the `DeepFrozen` vs `Unfrozen` distinction in Monte/OCapN, but enforced statically by Pony's type system rather than by a runtime auditor.

The system is called _deny capabilities_ because it denies certain combinations (mutable + shared + aliased) rather than granting positive permissions. Every reference starts with the full set of operations and capabilities are subtracted.

## Actor Model

Pony actors each have:
- A private heap (no shared mutable state)
- A message queue
- A behavior dispatch loop
- A scheduler-assigned thread

Communication is asynchronous message passing (`actor.behavior(args)`). Actors never share references to mutable data; they send `iso` references (transferring ownership) or `val` references (immutable copies). The type system enforces this at compile time.

Actor creation: `Actor.create(args)` returns a `tag` reference (unforgeable but non-capability in the ocap sense — it is not attenuable, though the actor can implement facets by dispatching on message type).

## ORCA: Distributed Garbage Collection Protocol

Pony's ORCA protocol is a _distributed reference-counting GC_ that can collect cycles across actors. ORCA stands for "ORCA isn't Really a Cycle-Aware..." — a recursive acronym acknowledging that it _is_ cycle-aware despite the joke.

### Architecture

Each actor has:
1. **Local GC**: A tracing GC (mark-sweep) over its own heap, independent of other actors.
2. **Remote reference map**: For each remote actor it references, the actor tracks a reference count (how many of its local objects point to that remote actor's objects).
3. **Message-based protocol**: Actors send GC protocol messages (`op:gc-exports`, `op:gc-answers` in CapTP terms) to coordinate collection.

### The Protocol (simplified)

1. **Per-actor RC table**: Actor A maintains an `rc(A -> B)` for each remote actor B it references — a count of local objects in A that hold references to objects in B.

2. **Block message**: When A needs to send a reference to B to a third actor C, it sends a "block" message that temporarily increments the count, preventing premature collection.

3. **Deferred decrement**: When A's `rc(A -> B)` drops to zero, A sends a "release" message to B, decrementing B's reciprocal count. B cannot be collected until all remote actors have released it.

4. **ORCA cycle detection**: An "orc" actor (one per group) periodically queries all actors in its group for their reference counts. Using a distributed mark-sweep variant over the count matrix, it detects cycles (e.g., A references B, B references C, C references A) and collects them when no external references exist.

### Comparison with OCapN/CapTP Distributed GC

| Dimension | Pony ORCA | OCapN CapTP |
|---|---|---|
| Cycle collection | Yes (distributed mark over RC matrix) | No (acyclic only; cycles handled per-vat) |
| Coordination | Dedicated "orc" actor per group | Pairwise `gc-exports`/`gc-answers` |
| Message overhead | Periodic global queries + per-update messages | Only on export/answer lifecycle events |
| Latency | Synchronization on orc queries | Decentralized, no global sync point |
| Portability | Tied to Pony runtime | Language-agnostic wire protocol |
| Complexity | High (cycle detection, orc, block messages) | Low (simple acyclic RC) |
| GC completeness | Full (local + distributed cycles) | Partial (distributed acyclic only) |

### Why CapTP Chooses Acyclic

CapTP's distributed GC is deliberately acyclic for several reasons the Pony comparison makes clear:

1. **Cross-language interop**: A cyclic distributed GC requires tight runtime integration (the orc protocol, block messages, deferred decrements). CapTP targets multiple languages; an acyclic protocol keeps the wire format simple and implementable in any runtime.

2. **Latency and coordination**: ORCA's cycle detection requires a global coordination point (the orc actor) and periodic queries. CapTP's pairwise protocol has no global state — each vat manages its own export/import tables independently.

3. **Cycle locality**: In practice, cross-vat cycles are rare (ocap discipline discourages them: a reference across a vat boundary is a capability, and cycles in capability graphs indicate design issues). Most objects form a DAG at the vat level. The acyclic assumption is pragmatic.

4. **Fallback to local GC**: A cross-vat cycle is not leaked forever in CapTP: each vat's local GC will eventually collect the proxy objects (far references), which send `gc-release` messages when collected, slowly unwinding the distributed cycle. It is not guaranteed timely, but it is safe.

## Relevance to OCapN Distributed GC

Pony's ORCA protocol is the most complete treatment of distributed GC in the actor-model literature. Its architectural choices illuminate CapTP's:

- **CapTP's simplicity is a feature, not a limitation.** Pony shows that full distributed cycle collection is possible but costly; CapTP's acyclic choice is the right one for a cross-language wire protocol.

- **The orc coordination pattern.** Prism's content-addressed core and deterministic replay could potentially replace the orc's coordination with a Merkle-rooted global snapshot — each actor's RC table is a deterministic function of its execution, so the distributed GC state is derivable from the program trace rather than requiring online coordination.

- **Reference capabilities as DeepFrozen enforcement.** Pony's `val` capability is a static `DeepFrozen`: the type system proves immutability before sending. Prism's type system could similarly enforce `DeepFrozen` as an effect-like constraint: a typeclass requiring no mutable cells reachable through the value, checked at compile time rather than at runtime like Monte's auditor.

- **iso as ownership transfer.** Pony's `iso` (isolated) maps to the OCapN notion of a "mint" or "brand" — a unique reference that, when sent, transfers exclusive authority. Prism's uniqueness typing (FBIP reuse) already tracks ownership for memory management; extending it to cross-vat `iso` transfer is a natural extension.

## Lessons for Prism

1. **Distributed acyclic RC is the pragmatic choice.** Prism should implement CapTP's `op:gc-exports`/`op:gc-answers` protocol, not ORCA. The acyclic wire protocol is simpler, interops with existing OCapN implementations, and matches the rarity of cross-vat cycles.

2. **Reference capabilities and effect rows are complementary.** Pony's deny capabilities control _aliasing_ of memory; Prism's effect rows control _authority_ over effects. They solve orthogonal problems and could coexist: `val` = `DeepFrozen` (immutable, copy-safe), effect row = set of ambient capabilities.

3. **Content-addressed orc replacement.** If Prism ever needs distributed cycle collection, its content-addressed core provides a novel alternative to Pony's online orc: RC tables are deterministic functions of the trace, so cycle detection becomes a post-hoc analysis over the Merkle DAG rather than an online protocol.

4. **Ownership transfer as capability attenuation.** Pony's `iso`-transfer is analogous to Prism's FBIP reuse: both prove that a value has no other live reference before moving/mutating it. The same analysis that enables in-place update could enable ownership transfer across vats, replacing copy-on-send with zero-copy handoff when the type system proves uniqueness.

5. **No ambient actor enumeration.** Unlike Erlang's `processes()`, Pony provides no way to enumerate actors — consistent with ocap discipline. Prism's concurrency should follow the same rule: vats are only reachable through references that were explicitly passed.
